# Tmux Pane Watcher Contract

Investigation pass for Phase 1.1 of `IMPROVEMENTS.md`. Documents the
behavior of every bash-watcher script generated by brain-mcp so that a
Node daemon can replace them without regressing observable behavior.

Source revision: branch `main` at HEAD on 2026-04-29 (uncommitted WIP
includes `--yolo` flag tweaks; watcher logic itself is committed).

## 1. Catalogue of generators

The repo contains **7** distinct bash-watcher script generators, not 4
as `IMPROVEMENTS.md` estimated. They cluster into **two contract
families**.

### 1a. Newer family — writes `STATE_FILE`, reconciled by Node finalizer

| # | File | Lines | Used for |
|---|---|---|---|
| 1 | `src/conductor.ts` | 387–411 | `brain-conductor` pi mode |
| 2 | `src/conductor.ts` | 437–461 | `brain-conductor` py mode |
| 3 | `src/conductor.ts` | 535–588 | `brain-conductor` claude mode |
| 4 | `src/index.ts`     | 1492–1545 | `brain_swarm_v2` (multi-CLI: claude/hermes/codex/pi) |
| 5 | `src/index.ts`     | 2531–2588 | `brain_wake` single-spawn (multi-CLI) |

All five share `attachTmuxWatcherFinalizer` (defined twice — at
`src/index.ts:183` and `src/conductor.ts:247` — with identical bodies).
On `watcher.on('exit')` the finalizer reads the state file:

| State file content | Reconciliation |
|---|---|
| `"timeout"` | `reconcileSessionExit(db, sid, 124, 'tmux watcher timed out')` |
| `"pane_closed"` or empty | `reconcileSessionExit(db, sid, 0, 'tmux pane closed')` |
| (no file) | no DB write — relies on agent's own pulse/markDone |

`watcher.on('error')` always falls back to `db.markDone(sid, -1, true, …)`.

### 1b. Older family — no STATE_FILE, no finalizer

| # | File | Lines | Used for |
|---|---|---|---|
| 6 | `src/tools/router-tools.ts` | 354–410 | router-spawned `brain_wake` (alt path) |
| 7 | `src/tools/swarm.ts`        | 372–424 | router-spawned `brain_swarm` (alt path) |

These differ only in the Node-side handlers and lack of state file:

- No `STATE_FILE` written from bash → on normal exit, **nothing
  reconciles the DB**. Session stays in whatever status the agent's
  last `pulse` left it. The watchdog (`src/watchdog.ts`) is the only
  thing that eventually notices.
- `watcher.on('error')`: router-tools calls `db.markDone(sid, -1, true,
  …)`; swarm calls `db.pulse(sid, 'failed', …)` — divergent.
- `watcher.on('exit')`: not handled.

This is a real correctness gap — an agent that hits the absolute
timeout via these two paths is killed (`/exit` then `kill-pane`) but
the DB row is not marked timed-out until watchdog sweeps it.

## 2. Watcher script anatomy

All seven scripts share the same skeleton. Differences are noted inline.

### 2.1 Inputs (templated into the script at generation time)

| Variable | Source | Notes |
|---|---|---|
| `TARGET` | `paneId` from `tmux split-window -P -F '#{pane_id}'` (or window name when `spawnLayout=window`) | targets are tmux pane IDs (`%17`) or window names |
| `PROMPT` | path under `tmpdir()` | only present in claude/multi-CLI variants |
| `BUFFER` | `brain-${ts}-${tmuxName}` | tmux paste-buffer name |
| `ABSOLUTE_TIMEOUT` | `config.timeout` (conductor) / `agentTimeout` (others) / hardcoded `3600` (`src/index.ts:1496`) | seconds; `0` means no timeout in some paths |
| `START_TIME` | `date +%s` at launch | |
| `STATE_FILE` | `tmpdir()/brain-watch-${ts}-${name}.state` | only newer family |
| `watcherFile` | `tmpdir()/brain-watch-${ts}-${name}.sh` | self-deletes |

### 2.2 Phase A — Ready detection (claude/multi-CLI only; absent in pi/py)

```
for i in $(seq 1 60); do
  sleep 2; check_timeout
  tmux display-message -t "$TARGET" -p "" || exit 0   # pane vanished
  CONTENT=$(tmux capture-pane -t "$TARGET" -p)
  if <readyPatterns> ; then READY=1; break; fi
  if <fallbackReady> ; then READY=1; break; fi
done
[ $READY -eq 0 ] && sleep 15
```

- Up to **120s** of polling (60 × 2s) followed by a 15s grace.
- Ready patterns are CLI-specific:
  - `claude`: `❯` (UTF-8 e2 9d af) OR `high effort | bypass perm | accept edits`.
  - `hermes`: `hermes | >> | ❯` OR `tools | model | ready`.
  - `codex`, `pi`/`py`: ready phase is skipped entirely (pi/py are
    non-interactive batch modes; codex inherits the claude variant
    text matching in `src/index.ts:2531+`).
- If pane disappears during ready phase → `exit 0` (no state file
  written → finalizer treats as `pane_closed`/exit-0 = success).

### 2.3 Phase B — Prompt paste (claude/multi-CLI only)

```
tmux load-buffer  -b "$BUFFER" "$PROMPT"
tmux paste-buffer -b "$BUFFER" -t "$TARGET"
sleep 0.5
tmux send-keys    -t "$TARGET" Enter
tmux delete-buffer -b "$BUFFER"
rm -f "$PROMPT"
```

Pi/py modes skip this entirely — the task is passed via argv/env when
spawning the pane.

### 2.4 Phase C — Liveness / timeout poll loop (all variants)

```
while true; do
  sleep 5
  check_timeout                                       # may exit 0 with "timeout"
  tmux display-message -t "$TARGET" -p "" || break    # pane closed → fall through
done
printf '%s\n' "pane_closed" > "$STATE_FILE"           # newer family only
rm -f "${watcherFile}"
```

**Pi/py simplified variant** (`conductor.ts:392+`, `437+`): `break`
falls through to the same `pane_closed` write. No ready/paste phases.

### 2.5 Timeout branch (`check_timeout`)

```
if [ $ELAPSED -ge $ABSOLUTE_TIMEOUT ]; then
  printf '%s\n' "timeout" > "$STATE_FILE"             # newer family only
  tmux send-keys -t "$TARGET" "${exitCmd}" Enter      # claude: /exit, hermes: /quit, pi/py: C-c
  sleep 5                                             # 2s for pi/py
  tmux kill-pane -t "$TARGET"
  rm -f "${watcherFile}"
  exit 0
fi
```

Exit-command differences:

| Mode | Soft exit cmd | Grace before `kill-pane` |
|---|---|---|
| claude / multi-CLI claude | `/exit` Enter | 5s |
| hermes | `/quit` Enter | 5s |
| pi / py | `C-c` | 2s |
| codex | `/exit` Enter (default branch) | 5s |

## 3. State machine summary

```
                   ┌────────────────────────────────────────────┐
                   │                                            │
   spawn-pane → READY (claude/multi-CLI only) → PASTED ─────────┤
       │            │  (60×2s + 15s grace)         │            │
       │            │                              │            │
       │            └─pane vanished→ exit 0 ───────┘            ↓
       │                                              ┌─→ POLL_RUNNING (5s tick)
       │                                              │       │
       └──────────────────────────────────────────────┘       │
                                                              │
                                  ┌──────────────────┬────────┘
                                  │                  │
                          (pane vanishes)        (timeout)
                                  │                  │
                          STATE = pane_closed   STATE = timeout
                                  │                  │
                                  └──────┬───────────┘
                                         ↓
                            watcher.exit → finalizer reads STATE_FILE
                                         ↓
                              reconcileSessionExit(db, …)
```

Older family (router-tools/swarm) skips the bottom two boxes — no state
file, no finalizer wiring on `exit`.

## 4. DB side-effects (newer family)

`reconcileSessionExit(db, sessionId, exitCode, detail)` in
`src/spawn-recovery.ts:416`:

1. `set_exit_code` — always.
2. If session already `done`/`failed` → bail (idempotent).
3. exit_code 0 + no confirmed work → `markDone(failed=true,
   "process exited before first heartbeat")`. **This is why short
   no-op spawns get marked failed even on clean exit.**
4. exit_code 0 + work observed → `markDone(failed=false, "completed
   (exit 0, …)")`.
5. exit_code != 0 → `markDone(failed=true, "process exited with code N:
   <detail>")`.

The watcher's two terminal states map to:

- `pane_closed` → `reconcileSessionExit(…, 0, 'tmux pane closed')` →
  outcome depends on `getSessionWorkSummary` heuristic.
- `timeout` → `reconcileSessionExit(…, 124, 'tmux watcher timed out')`
  → always `failed`.

## 5. Recovery / crash behavior

### 5.1 What survives a brain-mcp restart

- The bash watcher runs **detached** (`spawn('bash', […], {detached:
  true, stdio: 'ignore'})` + `unref()`). It survives the parent dying.
- The state file lives in `tmpdir()` — also survives.
- BUT: `watcher.on('exit')` only fires inside the parent that spawned
  it. If brain-mcp dies, the bash watcher will write the state file
  and then nobody reads it. `src/watchdog.ts:cleanupStaleTempFiles`
  removes `brain-watch-*` files older than 1h, with no reconciliation.
- → **A brain-mcp crash mid-flight loses the watcher's terminal
  signal.** Affected sessions sit until the watchdog reaps them as
  stale.

### 5.2 What watchdog provides as fallback

`src/watchdog.ts:62` keeps an in-memory `failureTracker` (lost on its
own restart — flagged in `IMPROVEMENTS.md` Phase 2.1). It polls every
15s, looks for sessions whose `last_seen` exceeds thresholds:

- < 30s old + dead → spawn-failure path.
- < 120s old + dead → crash path with respawn.
- > 300s with no pulse → ghost prune.

Liveness check at `watchdog.ts:103` shells out via `ps -o state= -p
$pid`; `spawn-recovery.ts:383` already uses `process.kill(pid, 0)`.
Two implementations of `isProcessAlive`. (`IMPROVEMENTS.md` 2.2.)

### 5.3 Files the watcher manages on tmpfs

- `brain-watch-${ts}-${name}.sh` — script body, deleted by watcher itself.
- `brain-watch-${ts}-${name}.state` — terminal-state, deleted by
  finalizer.
- `brain-prompt-${ts}-${name}.txt` — claude/multi-CLI only, deleted by
  watcher after paste.
- `brain-prompt-…` for conductor pi mode also deletes `systemFile`
  (note: pi cleanup deletes `${systemFile}` literally inside the
  template — `conductor.ts:406`).

## 6. Edge cases observed in source

1. **Pane closed during ready phase** (`exit 0` from inside Phase A):
   no state file written → finalizer treats as `pane_closed` → exit-0
   reconciliation. This relies on the empty-string branch in
   `attachTmuxWatcherFinalizer` (`raw === ''`).
2. **Window vs pane targets**: in router-tools/swarm,
   `spawnLayout='window'` sets `target = tmuxName` (a window name, not
   a pane id). All polling commands take `-t $TARGET`, which tmux
   accepts for both, but `tmux capture-pane -t windowname` captures
   only the active pane in that window — fine for ready detection but
   not robust if the window has multiple panes.
3. **`ABSOLUTE_TIMEOUT=0` semantics**: the newer family checks `[
   $ABSOLUTE_TIMEOUT -gt 0 ] && [ $ELAPSED -ge $ABSOLUTE_TIMEOUT ]`,
   so `0` disables the timeout. The older family in conductor.ts
   pi/py omits the `>0` guard — passing `timeout=0` would trigger
   immediate kill.
4. **Hardcoded 3600s** in `src/index.ts:1496` for swarm v2 — ignores
   any user-supplied `agentTimeout` for that path.
5. **5s polling cadence** is the minimum granularity for detecting
   pane closure. Daemon should hit ≤5s to match.
6. **Two `attachTmuxWatcherFinalizer` definitions** with identical
   bodies (one in `src/index.ts:183`, one in `src/conductor.ts:247`).
   Drift risk.

## 7. Inputs the daemon will need

Based on the catalogue, the daemon must be told (per pane) at least:

- `pane_id` (tmux pane id, e.g. `%17`).
- `session_id` (brain `sessions` row).
- `started_at`, `timeout_seconds` (0 = none).
- Optional `prompt_path` + `buffer_name` (only if it owns the paste step).
- `ready_markers` (list of substrings) + `fallback_markers` + `ready_strategy=skip|wait` (pi/py = skip).
- `exit_command` (`/exit` | `/quit` | `C-c`) and `kill_grace_seconds`.
- `cleanup_paths` (the `.sh`, `.state`, prompt file, optionally a system prompt file for pi mode).
- `finalizer_kind`: drives whether `markDone` semantics or
  `reconcileSessionExit` semantics apply (matters for the older-family
  callsites until they're migrated).

Outputs (per pane lifecycle event):

- on ready observed → optional event row (useful for metrics).
- on terminal → equivalent of `reconcileSessionExit(exitCode, detail)`
  with `exitCode ∈ {0, 124}` matching the current bash mapping.

## 8. Risks before writing the daemon

1. **Two contract families** — straightforward port keeps the older
   family's quirky `markDone` / `pulse('failed')` divergence. Either
   migrate router-tools.ts and swarm.ts to the newer reconciliation
   semantics in the same PR (recommended) or keep the divergence
   behind a `finalizer_kind` enum and migrate later.
2. **`exit 0` with no state file** is currently a feature (see edge
   case #1). Daemon must preserve "pane vanished during ready =
   success" — the simplest model is: any terminal where the daemon
   never observed timeout = `pane_closed`/exit-0.
3. **Pre-spawn race**: today the bash watcher is spawned *after*
   `tmux split-window` returns. Daemon takes a `pane_id` row inserted
   by the spawn path — if the daemon polls between insert and pane
   creation it must tolerate a brief "pane not yet alive" window. Use
   `started_at` + small grace before declaring missing.
4. **Process re-attachment after MCP restart** — daemon must, on
   start-up, replay open `pane_watches` rows: query each pane's
   `pane_dead`, treat absent panes as `pane_closed` exit-0, and resume
   watching the rest. This is currently impossible with the bash
   approach (state is in tmpfs only) so the daemon is strictly
   stronger.
5. **Pi mode's `systemFile` cleanup** — `conductor.ts:406` deletes
   `${systemFile}` (the agent system prompt) on `pane_closed` only.
   On timeout, it's leaked. Keep this asymmetric behavior or fix it
   in the same migration; flag explicitly.

## 10. Recommendation

The contract holds. Phase 1.2 (daemon design) can proceed with these
explicit decisions to make:

1. Migrate router-tools.ts/swarm.ts to the same
   `reconcileSessionExit`-based finalizer in the daemon PR (closes a
   real correctness gap), or keep the divergence behind a config
   field. **Recommend: migrate.**
2. Decide whether the daemon owns the prompt-paste step or whether the
   spawning code keeps pasting and only the polling moves into the
   daemon. **Recommend: daemon owns paste** so all tmux interaction
   is centralized.
3. Pick the IPC: SQLite `pane_watches` table + 5s daemon poll vs a
   short-lived UNIX socket for "wake daemon now". Plan currently says
   SQLite; the contract has no events that demand sub-second latency
   so SQLite-only is fine.
4. Preserve the `exit 0` = success edge (#1, #6.1) and the
   `ABSOLUTE_TIMEOUT=0 = no timeout` semantics.
